02 OS Environment — VM vs. Bare Metal

Your choice of environment has a direct impact on injection reliability. Understand the trade-offs before committing.

💻 Virtual Machine

Pros
  • Easy snapshots / rollback
  • Runs on any host OS
  • Disposable environment
Cons
  • USB passthrough unreliable
  • Injection may fail/drop
  • Channel hopping broken
  • Internal adapter inaccessible

⚙ Bare Metal RECOMMENDED

Pros
  • Full hardware access
  • Reliable injection
  • Native kernel drivers
  • Best performance
Cons
  • Dedicated machine or dual-boot
  • Less convenient for daily use

💾 Live USB Boot

Pros
  • No installation needed
  • Full hardware access
  • Portable
Cons
  • No persistence (unless configured)
  • Slower than bare metal install
  • Wordlists lost on reboot

06 Troubleshooting Common Issues

ERROR Interface not appearing after plugging in adapter
▶
Fix: adapter not detected
kali@kali:~$ dmesg | tail -20 # Look for USB attach messages and any driver errors kali@kali:~$ lsusb # If device shows in lsusb but no interface, try loading driver: kali@kali:~$ sudo modprobe 88XXau # For AR9271: kali@kali:~$ sudo modprobe ath9k_htc # RTL8812AU: plug into USB 3.0 port (blue port)
ERROR airmon-ng fails to enable monitor mode
▶
Fix: monitor mode failure
# Kill interfering processes first kali@kali:~$ sudo airmon-ng check kill # Try manual monitor mode as fallback kali@kali:~$ sudo ip link set wlan1 down kali@kali:~$ sudo iw dev wlan1 set type monitor kali@kali:~$ sudo ip link set wlan1 up kali@kali:~$ iwconfig wlan1 | grep Mode Mode:Monitor ← success
ERROR Packet injection test fails — "No answer from AP"
▶
Fix: injection test failure
# Lock to the AP's channel first kali@kali:~$ sudo iw dev wlan1mon set channel 6 # Then retry injection test kali@kali:~$ sudo aireplay-ng -9 wlan1mon Trying broadcast probe requests... Injection is working! # If still failing, try a powered USB hub or different port
ERROR Soft-blocked adapter — rfkill shows "blocked: yes"
▶
Fix: rfkill soft block
kali@kali:~$ rfkill list 1: phy1: Wireless LAN Soft blocked: yes kali@kali:~$ sudo rfkill unblock all kali@kali:~$ rfkill list 1: phy1: Wireless LAN Soft blocked: no Hard blocked: no

07 TX Power & Regulatory Domains

Your adapter's transmit power is capped by two things: the hardware maximum and the regulatory domain enforced by the kernel. Higher TX power extends your effective injection and capture range. In a lab environment you can adjust this.

⚠
Lab Only: Increasing TX power beyond your country's legal limit is illegal on public spectrum. Only do this in your isolated lab environment with no external interference.
kali@kali:~$ — Check and set TX power
# Check current regulatory domain kali@kali:~$ iw reg get global country US: DFS-FCC (2400 - 2483.5 @ 40), (N/A, 30), (N/A) # US = max 30 dBm on 2.4 GHz. Some regions allow more. # Set regulatory domain for lab (unconstrained in many drivers) kali@kali:~$ sudo iw reg set BO # BO = Bolivia — historically less restricted in out-of-tree drivers # Set TX power to 30 dBm (3000 mBm) kali@kali:~$ sudo iw dev wlan0mon set txpower fixed 3000 # fixed = don't let the driver reduce this automatically # 3000 = value in mBm (millidecibels-milliwatt); 3000 mBm = 30 dBm # Verify change kali@kali:~$ iwconfig wlan0mon | grep Tx-Power Tx-Power=30 dBm

08 Dual-Adapter Setup

Several attack modules require two wireless interfaces running simultaneously — one in monitor mode for injection/capture, and one in AP mode serving a rogue network. Evil Twin (Module 09) and Captive Portal (Module 10) both need this.

AttackInterface 1Interface 2
Evil Twin (Module 09)Monitor — deauth clientsAP mode — fake AP via hostapd
Captive Portal (Module 10)Monitor — recon/deauthAP mode — rogue portal
MITM with relayAP mode — client connectsManaged — upstream relay
kali@kali:~$ — Dual adapter setup
# Plug in both adapters, then list all interfaces kali@kali:~$ iw dev phy#0 Interface wlan0 type: managed ← adapter 1 (built-in or first USB) phy#1 Interface wlan1 type: managed ← adapter 2 (second USB) # Put wlan1 into monitor mode for injection/capture kali@kali:~$ sudo airmon-ng start wlan1 PHY Interface Driver Chipset phy1 wlan1 88XXau Realtek RTL8812AU (monitor mode enabled; use wlan1mon) # Verify both are available simultaneously kali@kali:~$ iw dev phy#0 Interface wlan0 type: managed ← AP mode / relay phy#1 Interface wlan1mon type: monitor ← injection + capture # Confirm wlan0 supports AP mode (required for hostapd) kali@kali:~$ iw phy phy0 info | grep -A 8 "interface modes" Supported interface modes: * AP ← must be present for hostapd * managed * monitor
💡
Rule: Never try to put one physical radio (phy) into two modes simultaneously. Each phy is a separate hardware radio. wlan0 and wlan0mon share the same phy#0 — you can only use one at a time on a single-radio adapter.
